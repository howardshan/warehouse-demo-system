# Southern Star Foods ERP Demo — 开发计划书（内部版）

> 版本：v0.1（草稿） · 日期：2026-07-22 · 作者：WMS 模块负责人
> 受众：CrunchyPlum 内部团队 · 语言：中文
> 代码库：`warehouse_demo/warehouse-demo-system`（WMS 与 Sales 同库开发）

## 1. 项目背景与目标

### 1.1 客户画像（2026-07 调研结论）

- **客户**：Southern Star Foods，高置信度为 Southern Star Distributors I, Ltd. 的商号（DBA），Plano, TX。⚠️ 待与客户当面确认实体身份。
- **实质业务**：小型 broadline 食品批发分销商（非狭义"餐厅设备商"），约 45 人、估算年营收 ~$28M，成立于 ~1987 年。
- **运营硬数据**（FMCSA USDOT 498144）：自营仓库 + 自有车队 **21 辆卡车 / 26 名司机**，跨州配送，货类含干货、生鲜、肉类、冷藏食品、饮料、纸制品。
- **客群（推断）**：独立餐厅/小连锁，固定路线每周配送。
- **现状痛点**：现用 ERP 已 20+ 年，功能落后、卡顿报错。仓库+车队是公司命脉，故对 WMS 格外看重。

### 1.2 Demo 目标

1. **赢得合作**：用可运行的 demo 证明我们理解食品分销业务，能力远超其旧系统。
2. **验证范围**：以 WMS + Sales 两模块为切入点，跑通"下单 → 拣货 → 装车 → 配送签收 → 退货"全链路。
3. **不追求完备**：demo 阶段明确"不做清单"（见 §3.3），管理客户预期。

### 1.3 成功标准

- 客户演示会上能流畅走完一条完整业务故事线（见 §7），无报错。
- 客户能直观感受到与旧系统的代差：实时告警看板、移动端作业、批次追溯秒级响应。
- 演示后客户愿意进入正式合同谈判（Discovery 阶段收费立项）。

## 2. 现状盘点

### 2.1 已有代码基础

`warehouse-demo-system`：Next.js 15 (App Router) + React 19 + TS + Tailwind 4 + Supabase（Postgres/Auth/RLS），约 12,100 行 TS/TSX + 3,600 行 SQL。40 张表 / 32 个迁移 / 75 个 Server Actions / 45 个页面 / 9 角色 RBAC / 中英西三语。文档体系完整（`docs/00-invariants.md` 14 条业务铁律 + 10 份 ADR）。

**结论：不重构，增量开发。** 下一个迁移序号从 `0033` 起。所有新功能遵循现有模式：领域纯函数进 `lib/domain/`，业务护栏下沉数据库触发器/RLS，页面走 Server Actions。

### 2.2 已覆盖的客户核心需求 ✅

| 能力 | 现有实现 |
|---|---|
| 盲收 + 三单核对 | receiving 四步流（盲收→送货单→发票→match） |
| 双单位 / catch-weight | products 双单位、weighing 称重页、weight-variance 域函数 |
| FEFO / 批次效期 | replenishment FEFO、临期告警瓦片、批次追溯（防环） |
| 信用控制 | 两次信用检查（SO 确认 + 装车前）、credit-control 页 |
| 多温区储位 | locations 带温区，两级储位（pick_face/reserve 等） |
| 配送与退货 | trips、POD 签收、return-notes/disposition、credit note 队列 |
| 盘点 | cycle_count_tasks + inventory_counts |
| 周转容器 | totes 表 |

### 2.3 缺口（demo 增量候选）❌

| # | 缺口 | 对客户的意义 | 归属 |
|---|---|---|---|
| G1 | **路线管理**：路线主数据、客户绑定路线+配送日、订单 cut-off、按送货顺序倒序装车 | 21 台车按固定路线跑，这是其日常运营核心 | WMS |
| G2 | **司机移动端 POD**：手机浏览器友好的趟次/签收/拒收界面 | 26 名司机的现场作业体验，最能"眼见为实" | WMS |
| G3 | **扫码拣选（RF 风格）**：移动端逐单指引 + 扫码/输码确认 | 替代纸单拣货，仓库工人核心体验 | WMS |
| G4 | **客户订货指南（order guide）+ 分客户定价** | 餐厅客户按历史常购清单快速下单 | Sales（与 WMS 有 ATP/单位换算接口） |
| G5 | 演示环境：缺 `.env.example`、未部署、文档滞后（写到 0017，实际 0032）、种子数据太少 | demo 能不能开起来的前提 | 共同 |
| G6 | 效期不足批次自动锁定 | 食品安全护栏的补强，演示加分项 | WMS |

## 3. Demo 范围

### 3.1 WMS 模块（本人负责）

- **P0（必做）**：G5 环境修缮与演示数据集、G1 路线管理、G2 司机移动 POD
- **P1（时间允许）**：G3 扫码拣选、G6 临期批次自动锁定
- **P2（演示打磨）**：演示故事线所需的 dashboard/数据完善

### 3.2 Sales 模块（同事负责，此处仅列边界假设）

- G4 order guide + 分客户定价、SO 录单体验优化、销售侧报表。
- 详见 §5 模块边界。

### 3.3 明确不做（对客户口径也如此）

- 财务开票/发票（沿用 ADR-0010"财务不做但数据必须存"）
- EDI 对接（850/810/856）
- 需求预测/自动采购建议
- 真实硬件 RF 枪对接（demo 用手机浏览器 + 摄像头/输码模拟）
- 路线优化算法（demo 只做手工排序，不做自动优化）

## 4. 分阶段开发计划（WMS 部分）

> 时间线未定，按阶段组织；每阶段独立可演示，客户会议定节奏后可直接映射到日历。工作量按 1 名开发（AI 辅助）估算。

### Phase 0 — 环境修缮与演示基础（约 3–4 人日）

| 任务 | 说明 |
|---|---|
| 补 `.env.example` | README 快速开始第一步依赖此文件，现缺失 |
| 部署演示环境 | Vercel + 云端 Supabase，执行 0001–0032 迁移 |
| 文档同步 | `docs/` 补 0018–0032 的迁移说明，跑通 `npm run docs:check` |
| 演示种子数据 v1 | 从现有 4 商品/3 客户扩充为贴近客户的数据集：约 60–100 SKU（干货/冷藏/冷冻/生鲜/纸品分类齐全，含 catch-weight 肉类）、15–20 家餐厅客户、3–5 条路线雏形 |

**出口标准**：任何团队成员 clone 后 15 分钟内本地跑起来；演示环境可访问。

### Phase A — 路线配送强化（约 6–8 人日）★ 核心差异化

| 任务 | 说明 |
|---|---|
| 迁移 0033+：`routes` 主数据 | 路线编号、配送日（周几）、默认司机/车辆 |
| 客户绑定路线 | customers 增加路线+配送日关联；订单 cut-off 时间（settings 可配） |
| 装车排序 | trips 升级：趟次关联路线，按送货顺序维护 stop 序列，装车清单**倒序**输出（后送先装） |
| 温区分舱提示 | 装车清单按干/冷藏/冷冻分组显示 |
| 触发器护栏 | cut-off 后订单进入下一配送日；趟次状态机校验 |

**出口标准**：能演示"周二线路 12 的 8 家客户，一键生成倒序装车清单"。

### Phase B — 移动端作业（约 6–8 人日）★ 最能打动客户

| 任务 | 说明 |
|---|---|
| 司机移动 POD | 手机版趟次页：今日路线 → 逐站签收/拒收/短装 → 拒收自动生成退货单与 credit 队列（复用现有 return-notes 链路） |
| 手写签名 | canvas 签名存 Supabase Storage，POD 详情可查 |
| 扫码拣选（P1） | 手机版拣货页：按温区分组逐行指引，摄像头扫条码或手输校验，接现有 recordPick/recordWeight |

**出口标准**：用一台真手机现场完成"拣货 → 签收"演示。

### Phase C — 演示打磨（约 3–4 人日）

| 任务 | 说明 |
|---|---|
| 演示故事线数据 v2 | 预置"演示日"状态：有在途 PO、待拣订单、在途趟次、一条临期批次告警 |
| Dashboard 核对 | 8 个告警瓦片在演示数据下全部有内容、不报错 |
| 临期批次自动锁定（G6） | 剩余效期低于阈值自动转隔离/锁定，触发器实现 |
| 演示脚本文档 | `docs/demo-script.md`：分角色的点击路径 + 台词要点 |
| 按角色 UAT | 9 角色逐一登录走查（README 自述的上线前置条件） |

**合计**：约 18–24 人日（不含 Sales 模块）。若客户要求 2–3 周快速演示，压缩方案 = Phase 0 + Phase A + 司机 POD（去掉扫码拣选与 G6）。

## 5. WMS / Sales 模块边界与协作约定（同一代码库）

### 5.1 归属划分

| 资产 | WMS（本人） | Sales（同事） |
|---|---|---|
| Server Actions | `warehouse.ts` `inventory.ts` `returns.ts` `repack.ts` `purchasing.ts` | `sales.ts` 及新增 order-guide/pricing actions |
| 页面 | receiving/stock/picking/weighing/shipping/trips/pod/returns | orders/approvals/customers 定价相关 |
| 共享表 owner | stock/batches/pick_lists/shipping_lists/trips/routes | sales_orders/so_lines/so_approvals/定价新表 |
| customers 表 | 路线绑定字段由 WMS 提交迁移 | 定价/order guide 字段由 Sales 提交迁移 |

### 5.2 协作机制

- **迁移序号排队**：在团队群里认领序号（0033、0034…）再动手，避免冲突；一个迁移文件只属于一个模块。
- **接口点（改动需双方过目）**：
  1. `confirmSalesOrder` / `revalidateSoGates` —— 信用/毛利/ATP 三道闸（Sales 调用，WMS 提供 ATP）；
  2. SO 冻结点 → 拣货单生成（`generatePickList`）；
  3. 分客户定价影响 SO 价格快照列，不得破坏"价格快照"铁律（invariants）；
  4. cut-off/路线绑定影响 SO 可选配送日（WMS 提供，Sales 消费）。
- **共同纪律**：改 `lib/domain/`、`docs/00-invariants.md`、RLS/触发器前先在群里说一声；`npm run docs:check` 与 `vitest` 过了才能合。
- **Git**：git 历史目前不可用（5 个 commit 全叫 "001"）。demo 期起强制规范：feature 分支 + 有意义的 commit message，建议尽快建远程仓库。

## 6. 技术方案要点

- 沿用现有架构，**零新框架引入**：Next.js Server Actions 即后端，业务护栏下沉 Postgres 触发器/RLS，领域纯函数进 `lib/domain/` 并配 vitest。
- 移动端不做原生 App：响应式 Web（手机浏览器），扫码用 Web API（`BarcodeDetector`，降级为手输）。正式合同阶段再评估 Flutter（公司既有能力）。
- 新页面继续走三语 `messages/*.json`（西语对德州仓库/司机是实用卖点，演示时可现场切换）。
- 新表一律带 RLS + 审计（沿用 audit_log 模式）。

## 7. 演示策略：一条订单的一生

演示会故事线（约 25 分钟，双人配合）：

1. **销售**（同事）：餐厅客户来电 → order guide 快速下单 → 信用闸拦截超额客户 → 审批放行
2. **仓库**（本人）：cut-off 后订单落入周二线路 → 生成拣货单 → 手机扫码拣选（含一条 catch-weight 肉类称重）→ 倒序装车清单
3. **司机**（本人，用真手机）：逐站配送 → 一站拒收两箱 → 现场签名 POD → 拒收自动进退货/credit 队列
4. **管理层视角**：dashboard 实时告警 + 任选一个批次演示"3 秒双向追溯"（对比旧系统卡顿报错，此处是情绪高点）
5. 收尾：明确"今天没演示的"（§3.3 不做清单）→ 引出正式合作的 Discovery 提案
