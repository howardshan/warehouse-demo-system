# 出货与签收

## 1. 业务规则
- 发运行必须与发运单、拣货行、订单行、商品和批次一致，并继承订单成交价、成本与称重属性快照。
- 装车放行执行信用检查②；被停供或 exposure 超限时，非财务/管理员不能放行。
- 只有已称重拣货单可放行；称重品离开待开票状态前必须有 `shipped_weight_lb`。
- 签收需要签收时间和签收人；签收后发运行禁止修改或删除。
- 签收时扣减实发库存、释放整笔实拣预留、关闭订单，并将待开票数据送入计费队列。

## 2. 涉及的表视图
- 表：`shipping_lists`、`sl_lines`、`pick_lists`、`pick_list_lines`、`sales_orders`、`so_lines`、`stock`、`credit_checks`。
- 视图：`v_billing_queue`、`v_credit_exposure`。

## 3. 对应的SQL文件
- `supabase/migrations/0014_shipping.sql`
- `supabase/migrations/0017_phase_rls_extras.sql`

## 4. 守护了哪些铁律
- 铁律 1、3：发运行继承成交快照，称重品按发运实重计费。
- 铁律 4：未称重不能放行，签收后明细锁死。
- 铁律 5、6：exposure 包含已签收未开票，并在装车放行前执行信用检查②。
- 铁律 10：签收关单并释放未发数量对应的预留，避免幽灵占用。

## 5. 为什么这么设计
- 放行时重新查信用，可覆盖订单确认后新增欠款或其他订单占额的变化。
- 签收作为库存、订单和计费的统一收口点，使三套状态在同一事务中前进。
- 计费采用视图队列而不在本系统生成发票，保留财务集成所需数据又不扩大边界。

## 6. 已知边界
- 系统只维护 `invoice_status` 与待开票队列，不生成正式发票或收款记录。
- 签收库存扣减使用下限 0 防负数；发现差异仍须走盘点或配送调整。
- `v_credit_exposure` 的开放订单与已签收未开票可能在业务过渡状态中重叠，接入财务系统前应核对口径。
